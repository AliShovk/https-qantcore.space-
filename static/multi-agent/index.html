💻 Примеры кода для трёх паттернов

👔 Role-based (CrewAI)
from crewai import Agent, Task, Crew

researcher = Agent(role="Researcher",
goal="Find trends")
writer = Agent(role="Writer",
goal="Write report")
team = Crew(agents=[researcher,writer],
tasks=[...], process=Process.sequential)
result = team.kickoff()
🧠 Supervisor (AutoGen)
from autogen import AssistantAgent

coder = AssistantAgent("coder")
reviewer = AssistantAgent("reviewer")
groupchat = GroupChat(
agents=[coder, reviewer], messages=[])
mgr = GroupChatManager(groupchat)
coder.initiate_chat(mgr,
message="Write a function...")
🔀 Graph (LangGraph)
from langgraph.graph import StateGraph

graph = StateGraph(State)
graph.add_node("think", think_node)
graph.add_node("act", act_node)
graph.add_conditional_edges(
"think", should_continue,
{"act": "act", "end": END})
graph.set_entry_point("think")
agent = graph.compile()

📖 Словарь: ключевые понятия

Agent — независимый LLM-процесс со своей ролью, инструментами и памятью.
Handoff — передача контекста от одного агента другому в зависимости от типа запроса.
State Graph — DAG-подобная структура с узлами обработки и conditional edges.
Tool Use — вызов внешних функций агентами: API, поиск, выполнение кода.
HITL — Human-in-the-Loop: человек проверяет решение до следующего шага.
Orchestrator — центральный агент, который распределяет задачи между worker agents.

🚀 Production-развёртывание multi-agent систем

Multi-agent системы требуют особого подхода к деплою: множество процессов, общее состояние, очереди сообщений. Вот проверенные production-паттерны:

🐳 Docker Compose
Каждый агент — отдельный контейнер. Общение через Redis/Kafka. Оркестратор управляет запуском и остановкой агентов. Подходит для малых и средних систем (3-10 агентов).
docker-compose up researcher analyst writer
☸️ Kubernetes
Каждый агент — Deployment с автоскейлингом. Состояние в StatefulSet (Redis/Postgres). Service Mesh (Istio) для observability. Для систем с 10+ агентами и variable load.
kubectl scale deploy/analyst --replicas=3
⚡ Serverless (AWS Step Functions)
Каждый шаг пайплайна — Lambda-функция. Состояние передаётся между шагами. Pay-per-use, автомасштабирование. Для событийно-ориентированных пайплайнов с редкими запусками.
Step: Research → Analysis → Write → Review

🧠 Memory в multi-agent системах: три уровня

Память — ключевой компонент любой multi-agent системы. Без неё агенты не помнят контекст между вызовами и не могут учиться. Три обязательных уровня памяти для production-систем:

💬 Level 1: Working Memory
Контекстное окно LLM. Хранит текущий диалог, инструкции, результаты последних шагов. Размер: 8K-200K токенов. Очищается при каждом новом запуске.
Инструменты: prompt engineering, context window management
📋 Level 2: Short-term Memory
Сессионная память. Redis/Postgres. Хранит промежуточные результаты агентов, статусы задач, логи шагов. Живёт в рамках одного пайплайна. Размер: MB-GB.
Инструменты: Redis, PostgreSQL, Memgraph
🗄️ Level 3: Long-term Memory
Векторная БД + граф знаний. Хранит выученные паттерны, успешные стратегии, ошибки. Переживает сессии и перезапуски. Ключ к самообучающимся системам.
Инструменты: ChromaDB, Pinecone, Neo4j, Weaviate
💡 Паттерн: Shared Memory Pool
Все агенты пишут в общий пул памяти (Redis pub/sub + векторную БД). Агент перед действием проверяет: «Кто-то уже решал похожую задачу?» — избегаем дублирования работы. Оркестратор читает пул для принятия решений о маршрутизации.

🔍 Observability: трассировка и отладка multi-agent систем

Агенты недетерминированы. Без observability вы не узнаете, почему система приняла то или иное решение. Три столпа наблюдаемости для multi-agent систем:

📋 Трассировка (Tracing)
Каждый вызов агента — span. Цепочка вызовов — trace. Видно: кто кому делегировал, сколько длился каждый шаг, какие инструменты использовались.
Инструменты: LangSmith, Arize Phoenix, OpenTelemetry
📊 Мониторинг (Metrics)
Счётчики: успешные/проваленные задачи, latency P50/P95/P99, расход токенов, стоимость. Алерты при аномалиях (latency вырос в 3×, success rate упал).
Инструменты: Prometheus, Grafana, Datadog
📝 Логирование (Logging)
Полный лог: промпты, ответы LLM, вызовы инструментов, ошибки. Нужно для отладки «почему агент сделал X а не Y». Хранить минимум 30 дней.
Инструменты: structlog, Loki, ELK Stack

🛡️ Безопасность и Human-in-the-Loop (HITL)

Multi-agent системы с доступом к файловой системе, сети и API — мощный вектор атаки. Архитектура безопасности должна быть заложена на уровне фреймворка:

✅ Обязательные практики
• Песочница: каждый агент в изолированном контейнере
• Read-only FS для агентов без явного разрешения
• Rate limiting на вызовы LLM (бюджет на задачу)
• Валидация вывода: schema check перед действием
• HITL для операций: delete, deploy, payment
⚠️ Распространённые уязвимости
• Prompt injection через пользовательский ввод
• Агент выполняет произвольный код из вывода LLM
• Утечка данных между сессиями агентов
• Неограниченные retry ≈ бесконечный счёт за токены
• Supply chain: вредоносный тул через MCP-сервер

💡 Продвинутые техники: RAG + Multi-agent

Комбинация Retrieval-Augmented Generation и multi-agent архитектуры открывает новые сценарии. Агенты с доступом к knowledge base работают точнее и требуют меньше ручного контекста.

📚 Research Agent + RAG
Агент-исследователь ищет в векторной БД релевантные документы, Analyst анализирует найденное, Writer синтезирует результат. В отличие от одиночного RAG — каждый этап выполняет специализированный агент.
🔄 Agentic RAG
Агент сам решает, когда искать в RAG, а когда использовать свои знания. Планирует поиск: формулирует query → ищет → проверяет результат → переформулирует если нужно → итерирует.
🔗 Graph RAG + Multi-agent
Граф знаний (Neo4j) + векторная БД. Агент навигатор обходит граф сущностей, агент-экстрактор извлекает факты, агент-верификатор проверяет достоверность. Применение: legal tech, scientific research.

🏗️ Архитектурные антипаттерны: чего избегать

❌ God Orchestrator
Один супер-агент делает всё: планирует, выполняет, проверяет. Проблемы: перегруженный контекст, узкое горлышко latency, невозможно отлаживать отдельные шаги.
✅ Решение: разделить на специализированных агентов с чёткими контрактами.
❌ Бесконечный диалог агентов
Агенты общаются без ограничений: A→B→A→B... Проблемы: экспоненциальный рост токенов, runaway costs, нет критерия остановки.
✅ Решение: max_rounds и convergence check после каждого раунда.
❌ Агент без валидации вывода
Агент генерирует JSON, следующий агент падает на парсинге. Или агент возвращает неполные данные, а система идёт дальше.
✅ Решение: Pydantic/Zod схемы + валидация между каждым шагом + retry.
❌ Общее состояние без блокировок
Два агента одновременно пишут в одну переменную состояния. Race condition, потеря данных.
✅ Решение: optimistic locking (Redis WATCH) или иммутабельное состояние (append-only log).
